# 04｜后端架构（Go + Gin）

## 1. 技术栈
- Go 1.22+
- Gin（HTTP 框架）
- GORM（ORM）
- PostgreSQL（主库）
- Redis（缓存/会话，后续接入）

## 2. 目录设计
```text
backend/
├─cmd/server/            # 程序入口
├─configs/               # 配置模板
├─internal/
│  ├─bootstrap/          # 启动装配（配置、DB、路由）
│  ├─config/             # 配置定义
│  ├─handler/            # HTTP handler
│  ├─service/            # 业务服务
│  ├─repository/         # 数据访问
│  ├─model/              # 数据模型
│  ├─middleware/         # 中间件
│  └─response/           # 统一响应结构
└─pkg/                   # 可复用公共包
```

## 3. 分层职责
- `handler`：参数校验、调用 service、组装响应；
- `service`：业务编排、领域规则；
- `repository`：数据 CRUD 与查询；
- `model`：实体定义与数据库映射；
- `middleware`：鉴权、日志、恢复、请求追踪。

## 4. 基础中间件（P0）
- Recovery：兜底 panic，防止服务崩溃；
- Request ID：每个请求分配唯一 ID；
- Logger：记录方法、路径、耗时、状态码；
- CORS：跨域策略（开发环境放宽）。

## 5. 模块优先级
1. Health 与系统基础能力
2. Auth（占位实现 + 接口约束）
3. Visit（就诊核心）
4. Medicine（库存）
5. Report（基础报表）

## 6. 演进策略
- P0 先单体服务，接口留版本号（`/api/v1`）；
- P1 根据压力再拆分 AI/消息为独立服务；
- 所有跨模块能力通过接口抽象，避免硬耦合。
