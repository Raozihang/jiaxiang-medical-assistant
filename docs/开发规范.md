# 开发规范

## 1. 通用原则
- 代码优先可读性与可维护性；
- 单次提交只做一类变更（功能、重构、文档分离）；
- 所有新功能需带最小可验证路径（可运行、可测试、可回归）。

## 2. 前端规范（React + TypeScript）

### 2.1 工程结构
- `src/app`：应用装配（路由、全局 Provider）；
- `src/modules`：业务模块；
- `src/shared`：跨模块复用能力；
- `src/pages`：页面组合层。

### 2.2 编码要求
- TypeScript 严格模式，不使用 `any` 逃逸；
- API 请求统一走 `shared/api/http.ts`；
- 组件命名 `PascalCase`，hooks 命名 `useXxx`；
- 页面状态显式处理：`loading`、`empty`、`error`。

### 2.3 质量门槛
```bash
npm run lint
npm run build
```

## 3. 后端规范（Go + Gin）

### 3.1 分层职责
- `handler`：参数校验、协议转换；
- `service`：业务编排与规则；
- `repository`：数据访问；
- `model`：数据实体映射。

### 3.2 编码要求
- 统一使用 `gofmt`；
- 错误必须返回并携带上下文语义；
- 接口响应统一结构：`code/message/data/request_id/timestamp`；
- 禁止在 handler 中写复杂业务逻辑。

### 3.3 质量门槛
```bash
go test ./...
go vet ./...
```

## 4. API 与数据规范
- API 前缀统一 ` /api/v1`；
- 字段命名优先 snake_case（与后端一致）；
- 破坏性接口变更必须升级版本路径；
- 数据库变更需同步更新 `docs/06-数据模型.md`。

## 5. 测试与验收
- 核心流程（登记、接诊、库存）必须可回归；
- Bug 修复必须附带复现路径与验证结果；
- 发布前至少完成一次端到端冒烟。

## 6. 安全与隐私
- 禁止硬编码密钥；
- 日志避免输出敏感个人信息；
- 医疗建议类输出必须保留“医生复核”环节。
